O67 -- i8 Rotate ISel via RLC/RRC -- feature test 44
=====================================================

Switches `ISD::ROTL` / `ISD::ROTR` for i8 from `Expand` to `Custom` and
introduces two target ISD nodes -- `V6CISD::ROTL8` and `V6CISD::ROTR8`
-- that map 1:1 to the 8080 `RLC` / `RRC` instructions via TableGen
`Pat<>` matchers. Constant-amount rotates lower into a chain of these
1-bit rotates, with the direction canonicalised to the shorter chain
(rotl by N == rotr by 8-N; pick whichever produces fewer ops; tie at
N=4 keeps the requested direction).

Implementation:
  V6CISelLowering.h      -- enum entries V6CISD::ROTL8, ROTR8;
                            LowerROTL/LowerROTR declarations.
  V6CISelLowering.cpp    -- ISD::ROTL/ROTR i8 -> Custom; LowerROTL,
                            LowerROTR; getTargetNodeName cases;
                            LowerOperation dispatch.
  V6CInstrInfo.td        -- SDT_V6CRot8 profile; V6Crotl8 / V6Crotr8
                            SDNodes; Pat<(V6Crotl8 Acc:$src), (RLC ...)>
                            and the symmetric RRC matcher.

Variable-amount rotates are vanishingly rare in real C; they fall back
to an i16-promotion synthesis ((x<<8)|x then SHL/SRL by amt, low byte).
No new runtime helper is needed.


## C source (tests/features/44/v6llvmc.c -- c8080.c is the same shape)

    typedef unsigned char u8;
    u8 rotl1(u8 x) { return (u8)((x << 1) | (x >> 7)); }
    u8 rotr1(u8 x) { return (u8)((x >> 1) | (x << 7)); }
    u8 rotl3(u8 x) { return (u8)((x << 3) | (x >> 5)); }
    u8 rotr3(u8 x) { return (u8)((x >> 3) | (x << 5)); }
    u8 rotl7(u8 x) { return (u8)((x << 7) | (x >> 1)); }
    u8 rotl4(u8 x) { return (u8)((x << 4) | (x >> 4)); }

    volatile u8 g_in = 0x5A;
    volatile u8 g_out;

    int main(int argc, char **argv) {
        (void)argc; (void)argv;
        g_out = rotl1(g_in);
        g_out = rotr1(g_in);
        g_out = rotl3(g_in);
        g_out = rotr3(g_in);
        g_out = rotl7(g_in);
        g_out = rotl4(g_in);
        return 0;
    }

`rotl4` is unchanged because the OLD path happens to canonicalise to
the same ADD-A-self chain (4 left == 4 right). Worst case is `rotl1`:
**~14x byte savings, ~14x cycle savings** on a single rotate.


## Where the OLD blowup came from

DAGCombine recognises `(x<<1)|(x>>7)` as `ISD::ROTL` and the generic
`Expand` action turns it back into `SHL i8 1 | SRL i8 7`. The i8 SRL
constant path zero-extends to i16 and runs the slow `V6C_SRL16`
expansion -- 7 instructions per shift step (`MOV A,H; ORA A; RAR;
MOV H,A; MOV A,L; RAR; MOV L,A`), times 7 steps for `>> 7`, plus the
SHL chain and the OR-of-halves. ~50+ instructions for one rotate.

## Notes

- We use RLC / RRC (circular rotates) rather than RAL / RAR (rotate
  through carry) because a chain of RLC/RRC has well-defined behaviour
  regardless of the previous CY value; RAL/RAR would require carry
  preservation across the chain. RAL/RAR remain available for future
  multi-byte shift work (e.g. an O68 follow-up that fixes i8 SRL/SRA
  by constant amount, the other half of the same blowup).
- No CLI flag: this is an unconditional ISel improvement -- the OLD
  Expand path is strictly worse, so there is no debug case where
  preserving it would be useful.
