# O65 ŌĆö MOV r, M + ALU r Fold (Direct Memory ALU Peephole)

*Lightweight post-RA counterpart to [O49](O49_direct_memory_alu_isel.md).*
*Where O49 proposes folding loads into ALU ops at ISel via dedicated*
*pseudos, this plan catches the remaining cases via a local peephole.*

### Scope (in order of implementation)

**Stage 1 ŌĆö strict adjacency**: `MOV r, M; OP r` where `r` is dead after
the peephole and no instructions sit between them (skipping only debug
MIs). Lowest-risk, highest-frequency variant.

**Stage 2 ŌĆö non-adjacent fold**: allow an arbitrary number of intervening
instructions between `MOV r, M` and `OP r`, gated on *all* of the
following for every instruction in between:
- does not read or write `r`,
- does not read or write any register that aliases HL (`H`, `L`, `HL`,
  and any super-register),
- does not read or write `A` or `FLAGS` (so both the A operand we'll
  feed into `OP M` and the FLAGS observer chain are unperturbed),
- is not a branch, call, return, or other control-flow boundary,
- is not `mayStore` (a store could alias `[HL]` and mutate the value
  we just loaded ŌĆö conservative veto, matches O49's reasoning).

**Stage 3 ŌĆö INR/DCR/MVI M triad (peephole backstop to O46/O49)**:
fold
```
MOV  A, M      ; V6C::MOVrM, dst=A, Uses=[HL]
INR/DCR A      ; V6C::INRr/DCRr, defs A+FLAGS
MOV  M, A      ; V6C::MOVMr, uses A+HL
```
into `INR M` / `DCR M`, and
```
MVI  A, imm
MOV  M, A      ; uses A+HL (A dead after)
```
into `MVI M, imm`. Requires proving HL is unchanged across the whole
three-instruction window (no intervening HL write, no intervening store
that could alias) ŌĆö same machinery as Stage 2. `A` dead after `MOV M, A`
is required for all three shapes so we don't accidentally observe the
modified accumulator.

## Problem

After register allocation, codegen routinely leaves sequences of the form:

```asm
LXI   HL, <addr>
MOV   r, M         ; 7cc, 1B ŌĆö load [HL] into scratch r
OP    r            ; 4cc, 1B ŌĆö A = A OP r  (or  CMP A with r)
```

The 8080 has direct `OP M` forms (opcodes 0x86/8E/96/9E/A6/AE/B6/BE) that
read the operand straight from `[HL]`:

```asm
LXI   HL, <addr>
OP    M            ; 7cc, 1B  (was 11cc, 2B ŌĆö saved 4cc, 1B)
```

O49 (direct memory ALU ISel) would catch this at DAG level via pseudos.
In practice, several cases arrive at post-RA unfolded because:

1. the load was materialized early and only later paired with an ALU op
   after other passes (copy-opt, peephole, branch shaping) rearranged code;
2. the load was generated by a lowering/expansion pass (e.g. spill reloads)
   that operates below DAG level and bypasses ISel patterns entirely;
3. O49 is not yet implemented.

A local peephole is cheap to build, orthogonal to O49, and recovers the
savings even when O49 ships ŌĆö any case O49 misses falls through to it.

### Example ŌĆö `tests/features/38/v6llvmc_new01.asm` (function `many_i8`)

```asm
LDA   __v6c_ss.many_i8+1
LXI   HL, __v6c_ss.many_i8
MOV   L, M                     ; L = [HL]
XRA   L                        ; A ^= L  (L dead after)
LXI   HL, __v6c_ss.many_i8+2
MOV   L, M
XRA   L
LXI   HL, __v6c_ss.many_i8+3
MOV   L, M
XRA   L
LXI   HL, __v6c_ss.many_i8+4
MOV   L, M
XRA   L
```

Each `MOV L, M; XRA L` pair becomes a single `XRA M`:

```asm
LDA   __v6c_ss.many_i8+1
LXI   HL, __v6c_ss.many_i8
XRA   M
LXI   HL, __v6c_ss.many_i8+2
XRA   M
LXI   HL, __v6c_ss.many_i8+3
XRA   M
LXI   HL, __v6c_ss.many_i8+4
XRA   M
```

Savings: **4 pairs ├Ś (4cc + 1B) = 16cc, 4B** in that function alone.

## Pattern

Fold:

```
MOV   r, M          ; V6C::MOVrM, defs r, uses HL
OP    r             ; V6C::{ADDr|ADCr|SUBr|SBBr|ANAr|XRAr|ORAr|CMPr}, uses A+r
```

into:

```
OP    M             ; V6C::{ADDM|ADCM|SUBM|SBBM|ANAM|XRAM|ORAM|CMPM}, uses A+HL
```

### Opcode mapping

| Register form | Memory form | Opcode |
|---|---|---|
| `ADDr` | `ADDM` | 0x86 |
| `ADCr` | `ADCM` | 0x8E |
| `SUBr` | `SUBM` | 0x96 |
| `SBBr` | `SBBM` | 0x9E |
| `ANAr` | `ANAM` | 0xA6 |
| `XRAr` | `XRAM` | 0xAE |
| `ORAr` | `ORAM` | 0xB6 |
| `CMPr` | `CMPM` | 0xBE |

All eight definitions already exist in
[V6CInstrInfo.td](../../llvm/lib/Target/V6C/V6CInstrInfo.td) lines 294ŌĆō303
with empty pattern lists ŌĆö no TableGen changes needed.

## Safety conditions

### Stage 1 ŌĆö adjacent `MOV r, M; OP r` (all required)

1. **Strict adjacency.** The two MIs are consecutive in the same MBB, with
   no instruction, label, or terminator between them. Skip over
   `DBG_VALUE` / `DBG_INSTR_REF` / `CFI_INSTRUCTION` only.
2. **Same register.** `MOVrM` destination == `OPr`'s RHS operand register.
3. **Source register excludes A.** `r Ōłł {B, C, D, E, H, L}`. `r = A` would
   mean `ADD A` (doubles A) ŌēĀ `ADD M` (adds memory to A). `MOV A, M` is
   itself a legal 8080 form but the fold isn't valid with `r = A`.
4. **`r` is dead after the OP.** Checked via `isRegDeadAfter()` (same
   helper used by `V6CXchgOpt` ŌĆö kill flag on the OP's use operand *or*
   no read below and not in any successor's livein).

   This single condition makes `r Ōłł {H, L}` safe: deleting `MOVrM` means
   HL is never clobbered, so `OP M` reads the original `[HL]`, exactly
   the value the OP was going to consume via `r`.
5. **HL not redefined between MOV and OP.** Guaranteed by condition (1)
   (adjacency ŌĆö the only instructions are the MOV and OP themselves,
   neither of which writes HL: `MOVrM` only writes `r`; register-form
   ALU only writes A + FLAGS).
6. **No flag dependency straddles the MOV.** `MOVrM` does not touch
   FLAGS, so there is nothing to preserve. (Condition noted for
   completeness.)

### Stage 2 ŌĆö non-adjacent `MOV r, M; ŌĆ” ; OP r`

Same as Stage 1 plus a `scanBetween(I, J)` walk over every MI strictly
between the MOV and the OP. For each MI `K`:

7a. `K` **does not read or write `r`** (including sub- and super-registers).
7b. `K` **does not read or write any register overlapping HL**
    (`H`, `L`, `HL`, or any super-register).
7c. `K` **does not read or write `A` or `FLAGS`** ŌĆö otherwise the A
    operand fed into `OP M` or the FLAGS observer downstream would be
    disturbed.
7d. `K` **is not a terminator, call, or control-flow boundary**
    (`isCall`, `isBranch`, `isReturn`, `isBarrier`, or
    `modifiesRegister(V6C::SP)` for conservative CALL-seq handling).
7e. `K` **is not `mayStore`** ŌĆö a store could alias the `[HL]` slot we
    just loaded. Conservative veto matching the O49 design reasoning.

Debug MIs are always skipped without imposing a condition.

### Stage 3 ŌĆö `MOV A, M; INR/DCR A; MOV M, A` and `MVI A, imm; MOV M, A`

Both shapes share the same `[HL]` slot on load and store, so HL must
be provably unchanged across the whole window:

8a. **All MIs in the window (exclusive of head and tail) satisfy
    Stage 2 conditions 7b/7d/7e.** `r` (= `A`) is expected to change by
    the middle MI, so 7a/7c are *not* required on the middle instruction
    of the `INR/DCR` form ŌĆö but 7c still applies to any extra MI inserted
    between the three core MIs by e.g. scheduling.
9a. **`A` is dead after `MOV M, A`** ŌĆö otherwise the caller observes a
    different A value (the pre-INR value was never reachable; `INR M`
    leaves A untouched).
10a. **Head and tail reference the same HL contents.** Trivially true in
     the zero-intermediate case; for the general case guaranteed by 7b
     (no HL writes) and 7e (no aliasing stores).
11a. **Middle operand register is A.** `INR/DCR A` / `MVI A, imm`.
     `INR r` for `r ŌēĀ A` is not foldable ŌĆö no `INR M` equivalent exists
     for non-A sources because the 8080 `INR M` always targets `[HL]`.

## Rewrite

### Stages 1 & 2 (`OP r ŌåÆ OP M`)

1. Build `OPM` MI with operands:
   * `def $A` (tied to `use $A` LHS, matching the existing `OPr` form),
   * `use $A` (the existing ALU LHS ŌĆö copy from original),
   * plus `implicit use HL` / `implicit def FLAGS` / `implicit def A` as
     per the `OPM` instruction description.
   For `CMPM` there is no `$A` def.
2. Insert `OPM` in place of the original `OPr`.
3. Erase `OPr`, then erase `MOVrM`. The intervening MIs (Stage 2) stay
   in place ŌĆö they were provably independent.

### Stage 3 (`INR/DCR/MVI M`)

`INR A` / `DCR A` / `MVI A, imm` + `MOV M, A` (with a matching `MOV A, M`
prefix in the INR/DCR case):

1. Build `INRM` / `DCRM` / `MVIM` with the same HL implicit use and
   FLAGS implicit def as their register-form counterparts (`MVIM` has
   no FLAGS def). `MVIM` carries the immediate from `MVI A, imm`.
2. Insert the new MI in place of `MOV M, A`.
3. Erase `MOV M, A`, the middle MI, and (INR/DCR form) the leading
   `MOV A, M`.

Kill flags:
* Transfer the earliest `MOVrM`'s HL kill flag (if any) onto the new
  memory-form MI's implicit HL use.
* The old `r` kill on the `OPr` use is irrelevant (register fully gone).
* No new defs are introduced beyond what the original sequence defined
  (A, FLAGS for INR/DCR; FLAGS unchanged for MVI).

## Where it lives

`V6CPeepholePass` (`V6CPeephole.cpp`) ŌĆö extend the existing post-RA
peephole pass with a new pattern handler. Placed before the self-MOV and
redundant-MOV patterns so the folded ALU becomes visible to any later
pass that scans for ALU shapes (e.g. `V6CZeroTestOpt`).

CLI toggle: reuses existing `-v6c-disable-peephole`.

### Pseudocode

```
for each MBB:
  I = MBB.begin()
  while I != MBB.end():
    if I is MOVrM with dst r:
      J = findAluConsumerFor(r, starting at next(I))
      if J found
         && r != V6C::A
         && isRegDeadAfter(r, J)
         && scanBetweenSafe(next(I), J, r):   // Stage 2 conditions 7a-7e
        emit OPM(A, A) before J, carry HL kill from I
        erase J  (old OPr)
        erase I  (MOVrM)
        I = next position
        Changed = true
        continue
    if I is MOVrM with dst=A  [Stage 3]:
      M = findMidInstr(A, starting at next(I))       // INR A / DCR A
      if M && next(M) is MOVMr with src=A
         && isRegDeadAfter(A, MOVMr)
         && scanBetweenSafe(next(I), M, V6C::A, allowAWrite=false)
         && scanBetweenSafe(next(M), MOVMr, V6C::A, allowAWrite=false):
        emit INRM/DCRM before MOVMr
        erase MOVMr, M, I
        continue
    if I is MVIr A, imm [Stage 3b]:
      K = findMovMr(A, starting at next(I))
      if K found && isRegDeadAfter(A, K)
         && scanBetweenSafe(next(I), K, V6C::A, allowAWrite=false):
        emit MVIM imm before K
        erase K, I
        continue
    ++I
```

A shared opcode-map helper (`aluRegToMemOpcode`, `incDecRegToMemOpcode`)
handles all nine `OP r ŌåÆ OP M` plus `INR/DCR/MVI` cases. The
`scanBetweenSafe(begin, end, r, allowAWrite=true)` helper encapsulates
Stage 2 conditions 7aŌĆō7e in a single walk.

## Relationship to O49

| Aspect | O49 (ISel pseudos) | O65 (this peephole) |
|---|---|---|
| Level | DAG / ISel | Post-RA peephole |
| Scope | All `OP M` forms | Stages 1ŌĆō3 cover all `OP M` (8 ALU) + `INR M` + `DCR M` + `MVI M, imm` |
| Register-allocator interaction | Eliminates the temporary entirely | Requires `r` dead after ŌĆö may miss cases |
| Risk | Higher (new pseudos, RA interaction) | Stage 1 very low; stages 2/3 low-medium (rely on explicit scan) |
| Coverage | Best possible | Backstop for the cases ISel missed |

Both can coexist: O49 removes the load at ISel when patterns fit; O65
mops up the residue. They target disjoint code paths in practice.

## Why this is safe where a naive fold is not

The O49 design doc correctly rejects an unconditional post-RA fold on
three grounds, addressed here:

1. *"Stale kill flags on the MOV destination ŌĆö deleting while live
   elsewhere causes silent bugs."* ŌåÆ Condition (4) uses a proper
   `isRegDeadAfter` liveness walk (not just the kill flag), identical
   to the mechanism `V6CXchgOpt` has used correctly since M8.
2. *"Cross-BB liveness isn't visible from a local scan."* ŌåÆ `isRegDeadAfter`
   checks block-end livens (successor liveins). Cross-BB visibility is
   exactly the information it was designed to surface.
3. *"The freed register may already hold something else."* ŌåÆ We don't
   "free" a register ŌĆö we delete an instruction that defined `r`. Nothing
   else *can* depend on that def because `r` is dead after. The RA's
   allocation of `r` elsewhere (before the MOV, or after the OP) is
   irrelevant.

## Estimated savings

- Stage 1 `OP r ŌåÆ OP M` fold: **4cc, 1B**.
- Stage 3 `MOV A, M; INR/DCR A; MOV M, A ŌåÆ INR/DCR M` fold: **8cc, 2B**
  per instance (was 20cc/3B, becomes 10cc/1B).
- Stage 3 `MVI A, imm; MOV M, A ŌåÆ MVI M, imm` fold: **5cc, 1B** per
  instance (was 15cc/3B, becomes 10cc/2B).
- Frequency: medium-high. Emerges naturally after O10 (static stack) and
  O20 (honest load/store defs) expose more single-use `[HL]` reads. The
  reference example (`v6llvmc_new01.asm`) shows 4 Stage 1 folds in one
  function body; Stage 3 triggers wherever a struct field or global byte
  is incremented/decremented/initialized in place.

## Implementation effort

Stage 1: ~40ŌĆō60 lines in `V6CPeephole.cpp`, one small opcode-map helper,
plus a lit test (`mov-alu-m-fold.ll`) covering each of the eight ALU
variants and at least one negative case.

Stage 2: +~30 lines for `scanBetweenSafe` helper + lit coverage of the
non-adjacent case (one intervening `MOV`, one failing case with a store
in between).

Stage 3: +~60 lines for `INR/DCR/MVI M` pattern matching, opcode map
extension, and `A`-dead liveness check, plus lit coverage of the three
shapes (INR M / DCR M / MVI M) and two negative cases (HL redefined
between the MOVs, A still live after store).

## Dependencies

None. Composes cleanly with every existing pass. Runs inside the
already-scheduled `V6CPeephole` pass so no new pipeline slot is needed.

## Risk

**Very low.** The fold is purely local, gated by proven-dead liveness,
and the target instruction encoding already exists and is exercised
elsewhere (e.g. inline asm, `O49` prototype work). Worst-case failure
mode is a missed fold, not miscompilation.
