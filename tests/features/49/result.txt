O15: Conditional Call Optimization

Purpose
-------
Fold the post-RA pattern

    Jcc  .Lskip
    CALL fn
  .Lskip:

into the equivalent single i8080 conditional CALL instruction
(CNZ / CZ / CC / CNC / CP / CM / CPE / CPO).

The optimisation is implemented in the `V6CBranchOpt` post-RA pass as
the `foldConditionalCalls` sub-pass. It runs after `invertConditionalBranch`
so that any inversion that re-shapes Jcc terminators is already done,
and before `removeRedundantJMP` / `foldConditionalReturns` so that
follow-on passes see the simplified CFG.

IPRA correctness
----------------
The fused conditional CALL keeps every operand of the original CALL
instruction, including the per-callee RegMask placed there by
`V6CTargetLowering::LowerCall`. The `V6CCallRegMaskVerifier` debug
pass continues to assert that every isCall instruction carries a
RegMask, so the narrowed clobber set produced by IPRA (O39) is
preserved without modification.

Bailouts
--------
The fold only fires when:
- The call basic block is reached via a Jcc-only fallthrough.
- The call basic block has a single predecessor (so it can be deleted).
- Its only non-debug instruction is V6C::CALL.
- It falls through to the same skip block targeted by the Jcc.

This explicitly excludes blocks that contain a return-value COPY
(test `cb_value_used`), argument-setup MIs, or branches to multiple
successors.

Build commands
--------------
Reference c8080 assembly:
  tools\c8080\c8080.exe tests\features\49\c8080.c -a tests\features\49\c8080.asm

V6C assembly verification:
  llvm-build\bin\clang -target i8080-unknown-v6c -O2 -S tests\features\49\v6llvmc.c -o tests\features\49\v6llvmc_new01.asm

Assembly line counts
--------------------
c8080.asm:          230
v6llvmc_old.asm:    145
v6llvmc_new01.asm:  145

Note: tests/features/49 was created after the implementation patch, so
v6llvmc_old.asm is a current compiler snapshot rather than a pre-O15
historical baseline. The relevant comparison is between the c8080
output (which already emits conditional CALLs) and the V6C output,
both of which now contain CZ / CNZ / CC / CNC / CP / CM forms instead
of separate Jcc + CALL pairs.

Conditional CALL coverage in v6llvmc_new01.asm
----------------------------------------------
- cb_eq        ORA A      / CZ  notify
- cb_ne        ORA A      / CNZ notify
- cb_ult       SUB/SBB    / CNC notify
- cb_uge       SUB/SBB    / CC  notify
- cb_slt       SUB/SBB    / CP  notify
- cb_sge       SUB/SBB    / CM  notify
- cb_value_used (negative): keeps `JZ ... ; CALL produce` because the
  return value is consumed (call block contains a result COPY).

Representative output (V6C)
---------------------------
cb_eq:
  ORA A
  CZ  notify
  LDA observed
  INR A
  RET

cb_ult:
  MVI A, 0x63
  SUB L
  MVI A, 0
  SBB H
  CNC notify
  LDA observed
  INR A
  RET

cb_value_used:                    ; negative case — value is used
  ORA A
  JZ  .LBB8_2
  CALL produce
  JMP .LBB8_3
.LBB8_2:
  MVI A, 7
.LBB8_3:
  LXI H, observed
  ADD M
  RET

Reference c8080 emission
------------------------
cb_eq (i8080 mnemonics):
  ORA A
  CZ  notify
  LDA observed
  INR A
  RET

cb_ult:
  LXI D, 0xFF9C        ; 65436 = -100
  DAD D
  CNC notify
  LDA observed
  INR A
  RET

Both compilers use the conditional CALL opcodes, confirming O15 brings
V6C output to parity with c8080 on this pattern.

------------
