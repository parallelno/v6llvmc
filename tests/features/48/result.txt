O69: Direct Frame-Index Memory Pseudos
=======================================

Purpose
-------
Exercise direct stack-relative i8/i16 loads and stores selected before
register allocation:

  - V6C_LOAD8_FI
  - V6C_LOAD16_FI
  - V6C_STORE8_FI
  - V6C_STORE16_FI

The "old" baseline is produced with the O69 selector hooks
(`ISD::LOAD`/`ISD::STORE` cases in `V6CISelDAGToDAG.cpp`) temporarily
disabled so that frame-index loads and stores fall back to the prior
`V6C_LEA_FI` + `V6C_LOAD*_P` / `V6C_STORE*_P` shape.

Build commands
--------------
Reference c8080 assembly:
  tools\c8080\c8080.exe tests\features\48\c8080.c -a tests\features\48\c8080.asm

V6C with O69 disabled:
  llvm-build\bin\clang -target i8080-unknown-v6c -O3 -S \
      tests\features\48\v6llvmc.c -o tests\features\48\v6llvmc_old.asm \
      -mllvm -mv6c-annotate-pseudos \
      -mllvm -v6c-disable-alloca-promote \
      -mllvm -v6c-disable-static-stack-alloc

V6C with O69 enabled:
  llvm-build\bin\clang -target i8080-unknown-v6c -O3 -S \
      tests\features\48\v6llvmc.c -o tests\features\48\v6llvmc_new01.asm \
      -mllvm -mv6c-annotate-pseudos \
      -mllvm -v6c-disable-alloca-promote \
      -mllvm -v6c-disable-static-stack-alloc

C test case
-----------
typedef unsigned char u8;

volatile u8 g8;
volatile int g16;

__attribute__((noinline))
u8 load8_stack_arg(u8 a0, u8 a1, u8 a2, u8 a3,
                   u8 a4, u8 a5, u8 a6, u8 a7) {
    return a7;
}

__attribute__((noinline))
int load16_stack_arg(int a0, int a1, int a2, int a3) {
    return a0 + a1 + a2 + a3;
}

__attribute__((noinline))
u8 store8_local(u8 x) {
    volatile u8 slot;
    slot = x;
    return slot;
}

__attribute__((noinline))
int store16_local(int x) {
    volatile int slot;
    slot = x;
    return slot;
}

int main(int argc, char **argv) {
    (void)argc; (void)argv;
    g8  = load8_stack_arg(1, 2, 3, 4, 5, 6, 7, 8);
    g16 = load16_stack_arg(1, 2, 3, 4);
    g8  = store8_local(g8);
    g16 = store16_local(g16);
    __builtin_v6c_out(0xED, g8);
    __builtin_v6c_out(0xED, (unsigned char)g16);
    __builtin_v6c_hlt();
    return 0;
}

    XCHG

load8_stack_arg:
    ;--- V6C_LOAD8_FI ---
    LXI  H, 2
    DAD  SP
    MOV  A, M
    RET

load16_stack_arg:
    ;--- V6C_ADD16 ---

    DAD  B
    XCHG
    ;--- V6C_LOAD16_FI ---

    DAD  SP
    MOV  C, M
    INX  H
    MOV  B, M
    XCHG
    DAD  B
    RET

store8_local:
    LXI  H, 0xffff
    DAD  SP
    SPHL
    ;--- V6C_STORE8_FI ---
    LXI  H, 0
    DAD  SP
    MOV  M, A
    ;--- V6C_LOAD8_FI ---

    DAD  SP
    MOV  A, M
    LXI  H, 1
    DAD  SP
    SPHL
    RET

store16_local:
    PUSH PSW
    MOV  B, H
    MOV  C, L
    ;--- V6C_STORE16_FI ---
    LXI  H, 0
    DAD  SP
    MOV  M, C
    INX  H
    MOV  M, B
    ;--- V6C_LOAD16_FI ---
    LXI  H, 0
    DAD  SP
    MOV  E, M
    INX  H
    MOV  D, M
    XCHG
    POP  PSW
    RET

Notes
-----
- load8_stack_arg: OLD used `LXI H,2; DAD SP; XCHG; LDAX D` (LEA via HL,
  hand-off to DE, then LDAX). NEW collapses to `LXI H,2; DAD SP; MOV A,M`,
  removing the XCHG and the DE temporary entirely.
- load16_stack_arg: OLD selected BC for the FI address, forcing a
  `MOV B,H; MOV C,L` copy and a 5-instruction `LDAX B / INX B / LDAX B`
  load chain. NEW reads directly into BC through the M operand
  (`MOV C,M; INX H; MOV B,M`) and adds a single XCHG to swap BC into
  position for `DAD B`.
- store8_local: OLD held the FI address pinned in DE across both the
  store and the reload. NEW emits two independent FI sequences
  (`LXI;DAD SP` twice) because each direct FI pseudo recomputes the
  address. Single-use is faster, but two uses against the same slot
  cost +3B/+6cc here. This is the pseudo's expected trade-off and
  documented in the future plan: the address temporary is removed
  pre-RA at the cost of duplicated address arithmetic if RA decides
  not to materialize a separate LEA.
- store16_local: NEW removes the `STAX/LDAX B` round-trip and uses
  direct `MOV M,C/B` + `MOV E,M / MOV D,M` over HL. One XCHG remains
  to deliver the loaded value into HL for the return. Net -1B/-5cc.
- main is identical in both builds (no frame-index loads/stores in
  main itself; the volatile globals route through V6C_LOAD16_G /
  V6C_STORE16_G).
